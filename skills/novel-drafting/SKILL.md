# 正文起稿

## 功能定位

按最小上下文包稳定产出正文，避免长篇写作时把整库资料一次性塞进上下文。正式起草前，先用写前自检锁定本章任务、冲突、前后照应、人物状态和章尾推进器。写作完成后，要把状态回写到项目文件里；写长篇和写短篇时，节奏目标不能混用。

这个 skill 的重点不是“尽快把字数铺出来”，而是先保证戏成立、逻辑闭合、细节落地、章尾有效，再去追求篇幅。默认把“质量稳定可持续”放在“当下写得多”前面。

## 最小上下文包

默认只加载：

1. `00-书核/作品总表.md`
2. `00-书核/立项单.md`
3. 当前章节 beat
4. 当前卷文件

按需追加，而不是默认全读：

- 相关角色文件
- 相关 canon 条目
- 上一章正文
- `90-运行/当前进度.md`
- 相关拆文分析中的“因果怎么闭合”
- 相关拆文分析中的“细节抓手”
- 相关拆文分析中的“迁移到我项目里的注意点”

如果是短篇，再优先读：

- `05-市场/趋势笔记.md`
- 目标短篇对标的拆文结果

如果是长篇，再优先读：

- `20-大纲/因果/场景因果图.md`
- `20-大纲/回收/回收总账.md`
- `90-运行/连载驾驶舱.md`

在上下文包选完后，最后读取 `references/写前检查表.md`，把空项补到“可以下笔”的程度再起草。

如果是连续日更、周更或长篇续写，再补读：

- `references/连载续写卡.md`

如果这次重点是提高正文成书感，再补读：

- `references/章节质量标准.md`

## 写前锁定

起草前至少明确：

1. 这章或这场到底要完成什么任务
2. 这一段是被什么前情结果逼出来的
3. 角色这次要主动做什么，以及会撞上什么阻力
4. 本章要碰哪条旧线、埋哪条新线、回收哪条前文信息
5. 准备用哪些具体细节承载信息和情绪，而不是写到一半全靠抽象解释
6. 结尾要留下什么继续阅读的推动力

如果以上有三项以上说不清，先回到 `novel-outlining` 或连续性文件补齐，不要硬写。

写前还要额外锁一件事：这一章读者读完后，最应该留下哪一种感觉或哪一个问题。没有这个“读后效应”，就很容易写成信息齐全但阅读驱动力不足的平章。

## 起稿执行链

默认按这个顺序起稿，不要一上来就平铺正文：

1. 先锁本章最重要的一场戏
2. 先决定角色这次主动做什么，而不是先写背景说明
3. 先把冲突和阻力摆上桌，再补环境和解释
4. 先让局面发生变化，再回收人物感受
5. 先保章尾发动机，再看是否需要加一层情绪余震

如果写到一半发现正文开始用解释句硬推，先停下来，回头看写前卡和章节质量标准，不要靠继续加字数掩盖问题。

## 场景级写作循环

每场至少要完成：

1. 角色想做成什么
2. 被什么人或什么事卡住
3. 这一场之后局面变成了什么
4. 这一场给后面留下了什么债

如果参考了拆文结果，再额外自检：

5. 这场的因果是否闭合
6. 这场有没有接上前文埋线或人物状态
7. 这场用了什么具体细节承载信息或情绪

长篇正文再额外自检：

8. 这一场有没有把主角、关系或悬念往前真正推一步
9. 这一场是否留下了下一场非写不可的直接后果

章节结束时保留明确的推动力，不要只是平静收束。

短篇场景则优先保证每场都在逼近题眼或结局效果，不追求长篇式层层留钩。

## 质量底线

起草时默认守住这些底线：

1. 不用整段总结代替戏
2. 不用作者解释代替人物行动
3. 不用抽象情绪词代替身体反应和关系变化
4. 不让角色在没有代价的情况下突然想通、突然变强、突然改变立场
5. 不让章尾只做平静收束，至少要留下问题、压力、决定或余震中的一种

如果一章同时踩中三条以上，优先视为需要重写结构，而不是只修句子。

## 写后立刻自检

正文初稿完成后，先不要急着润色，先用下面顺序自检：

1. 读 `references/写后自检卡.md`
2. 看本章是否已经达到 `references/章节质量标准.md` 的最低标准
3. 如果主要问题是前后打架、漏线、状态没续上，转 `novel-checkup`
4. 如果主要问题是段落空、转折站不住、章尾没力，转 `novel-revision`

## 回写要求

写完正文后至少更新：

- `30-正文/第X卷-卷名/第XXX章.md` 中对应章节
- `90-运行/当前进度.md` 中的当前进度、未决问题、下一步
- 如出现新 hard canon，补到 `10-设定/硬设定.md` 或记入 `90-运行/决策记录.md`
- 如本章推进了伏笔、人物状态、因果债或关系债，联动更新 `20-大纲/因果/场景因果图.md` 或 `20-大纲/回收/回收总账.md`

## 不在这里完成的事

- 不在 drafting 阶段大规模重构整本书结构
- 不在 drafting 阶段做精修级润色
- 如果发现大纲失效，转回 `novel-outlining`
- 如果发现成稿需要重点修订，转到 `novel-revision`
- 如果只是去 AI 味，不在这里硬修，转到 `novel-deslop`
- 如果核心诉求是“先检查这一章前后顺不顺、有没有漏线”，先转到 `novel-checkup`

需要更细的加载规则时，读取 `references/上下文包.md`；需要具体写前检查项时，读取 `references/写前检查表.md`；需要提高章节质量时，读取 `references/章节质量标准.md`；需要做长篇续写防漂时，读取 `references/连载续写卡.md`；需要写后自检时，读取 `references/写后自检卡.md`。
