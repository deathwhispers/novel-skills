# 读者反馈机制

> **使用说明**：长篇连载中后期，"读者反馈"是重要的"数据源"——帮你判断哪里好哪里坏。
> 本表是"读者反馈的收集 + 分析 + 行动"机制。
> 配 `novel-quality` 使用——体检是"内部诊断"，本表是"外部数据"。

## 一、读者反馈的 3 大来源

| 来源 | 实时性 | 关键作用 |
|------|------|---------|
| **评论** | 实时 | 读者当下反应 |
| **追读数据** | 延迟 | 读者流失点 |
| **段评/书评** | 延迟 | 深度反馈 |

## 二、评论的 5 大类型

> 读者评论是"最实时的反馈"——分类整理很重要。

| 类型 | 含义 | 处理 |
|------|------|------|
| **夸** | 读者说"好看"/"爽" | 继续这种 |
| **骂** | 读者说"烂"/"水文" | 反思但不全信 |
| **建议** | 读者提意见 | 部分采纳 |
| **讨论** | 读者分析剧情 | 看他们对伏笔/反转的反馈 |
| **催更** | 读者说"快更" | 维持节奏 |

## 三、追读数据的 3 大核心指标

| 指标 | 含义 | 阈值 |
|------|------|------|
| **追读率** | 上一章读者中下章继续看的比例 | < 70% 警告，< 50% 危险 |
| **章节读完率** | 读者是否读完本章 | < 60% 提示章尾弱 |
| **弃书率** | 上一章读者中永远不看的比例 | > 30% 提示节奏问题 |

> **关键**：追读数据是"硬数据"——比评论更真实。

## 四、读者反馈的"分类记录表"

> 每章发布后记录关键反馈。

| 章节 | 主要反馈 | 数据 | 我的应对 |
|------|--------|------|---------|
| Ch 1 | "节奏好""想看后续" | 追读 85% | 继续 |
| Ch 2 | "李昭太坏了" | 追读 78% | 继续 |
| Ch 5 | "中段有点慢" | 追读 60% | 加快节奏 |
| Ch 10 | "苏晚好可爱" | 追读 75% | 继续 |
| ... | | | |

## 五、读者反馈的 5 大分析维度

| 维度 | 分析什么 | 用法 |
|------|--------|------|
| **剧情** | 哪些剧情被夸/被骂 | 继续/调整 |
| **角色** | 哪些角色被喜欢/被讨厌 | 加强/调整 |
| **节奏** | 哪些章节读者觉得快/慢 | 调整 |
| **伏笔** | 哪些伏笔被讨论/被期待 | 加强/揭晓 |
| **感情线** | 感情线被支持/反对 | 调整 |

## 六、读者反馈的"5 个不能全信"

| 不能全信 | 原因 |
|--------|------|
| **个别读者的极端反馈** | 1 个读者不代表所有 |
| **早期读者的反馈** | 后期读者可能完全不同 |
| **付费读者的反馈** | 可能偏向"看广告" |
| **白嫖读者的反馈** | 可能偏向"看免费" |
| **粉丝的反馈** | 可能"护主" |

> **核心规则**：**数据 > 个别反馈**。追读率比评论更真实。

## 七、读者反馈的"分类行动"

> 不同反馈有不同应对。

| 反馈 | 行动 |
|------|------|
| **追读率突降** | 立刻做卷体检/长线扫描 |
| **某角色被骂"工具人"** | 给这个角色加弧线 |
| **某剧情被骂"突兀"** | 加铺垫/调整 |
| **某伏笔被催"快揭晓"** | 评估是否提前 |
| **某感情线被反对** | 评估是否继续 |
| **某节奏被嫌"太慢"** | 加快/加小高潮 |
| **某节奏被嫌"太快"** | 加过渡 |

## 八、读者反馈的"工具"

| 工具 | 用途 |
|------|------|
| **评论分类表** | 整理读者反馈 |
| **追读数据表** | 追踪数据变化 |
| **建议箱** | 收集建议 |
| **段评精华** | 收集高质量讨论 |
| **数据周报** | 周度数据汇总 |

## 九、读者反馈的"工作流"

### 每周反馈处理

| 时段 | 时长 | 任务 |
|------|------|------|
| 每天 | 10 分钟 | 看评论+分类 |
| 周末 | 30 分钟 | 整理反馈表+数据汇总 |
| 周末 | 30 分钟 | 反思"这周哪些反馈要行动" |
| 下周 | - | 行动反馈（修改大纲/卷纲/节拍） |

### 每月反馈处理

| 时段 | 时长 | 任务 |
|------|------|------|
| 月初 | 1 小时 | 整理月度反馈 |
| 月初 | 1 小时 | 分析数据趋势 |
| 月初 | 1 小时 | 反思+调整 |

## 十、读者反馈的"AI prompt"

### Prompt：分析评论

```
请按以下信息，分析最近一周的读者评论。

【评论列表】
{{}}

【分析维度】
1. 主要夸什么？
2. 主要骂什么？
3. 主要建议什么？
4. 哪些反馈是真实的？
5. 哪些反馈是个别的？
6. 哪些反馈需要立即行动？

【输出】
1. 反馈分类（夸/骂/建议/讨论/催更）
2. 主要趋势
3. 建议行动
4. 不需行动的反馈（避免被少数声音带偏）
```

### Prompt：分析追读数据

```
请按以下信息，分析最近 10 章的追读数据。

【数据】
- 章节 1 追读率：{{}}%
- 章节 2 追读率：{{}}%
- ...
- 章节 10 追读率：{{}}%

【分析维度】
1. 哪些章节追读率突降？
2. 哪些章节是拐点？
3. 整体趋势是升/降/平？
4. 与上个月的对比？
5. 推测原因

【输出】
1. 数据总结
2. 问题章节列表
3. 建议行动
```

## 十一、读者反馈的"边界"

> 读者反馈是参考，不是命令——不能完全被读者左右。

| 边界 | 说明 |
|------|------|
| **不能改主线** | 主线是核心，不能为反馈而改 |
| **不能迎合所有人** | 满足一部分读者即可 |
| **不能完全不理** | 集中反馈要重视 |
| **不能改结局** | 结局是规划好的 |
| **不能为反馈写** | 写你自己的故事 |

> **核心规则**：**写自己相信的故事，听读者的声音但不盲从。**

## 十二、与读者反馈改写决策.md 的关系

> 反馈机制的十一章解决"收集+分析"，**分析完之后怎么改**见 `读者反馈改写决策.md`。

| 本文件负责 | 读者反馈改写决策.md 负责 |
|-----------|------------------------|
| 收集分类 | 分析结果 → 改哪几章 |
| 追读数据分析 | 改写指令（Prompt A/B/C/D）|
| 每周/每月工作流 | 执行链路（回 Step 3/4/2）|
| 反馈的 5 个不能全信 | 决策纪律（不能读者骂什么改什么）|

**完整反馈闭环**：
> 收集分类（机制） → 追读分析（机制） → 归因分析（改写决策 Prompt A） → 改哪几章（改写决策 Prompt B/C/D） → 执行链路 → 追读验证 → 记录日志
