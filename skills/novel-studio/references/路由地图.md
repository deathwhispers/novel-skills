# 小说任务路由地图

## 先看用户要什么

| 用户当前意图 | 路由 |
|---|---|
| 写章节、场景、短篇或探索人物声音 | `novel-writing` |
| 检查、重写、润色、发布前精修 | `novel-quality` |
| 主线、卷纲、节拍、伏笔与回收 | `novel-outline` |
| 人物、世界规则、地点、时间线 | `novel-worldbuilding` |
| 新建项目、立项、导入旧稿 | `novel-project` |
| 读者评论、追读和反馈调整 | `novel-feedback` |
| 平台、题材、对标和商业包装 | `novel-market` |

## 写作任务分流

用户要求正文时，先判断模式：

- 商业连载：突出承诺、兑现和持续阅读动力。
- 类型长篇：突出类型体验、因果、悬念和人物弧线。
- 文学叙事：突出观察、情绪真实、语言与主题回响。
- 短篇：突出压缩、结构回声、意象和结尾。
- 探索起草：先发现人物声音或故事可能性，不用完整大纲阻止试写。

模式能从项目和正文判断时直接执行。只有模式差异会显著改变成文方向时，最多追问一个问题。

## 多步请求

- “检查并修订”：一次进入 `novel-quality`，先按授权诊断，再修订。
- “续写并检查”：先 `novel-writing` 输出正文；只有用户要求或发现硬伤时进入 `novel-quality`。
- “从点子开始并试写”：用 `novel-project` 建最小书核，然后允许 `novel-writing` 探索起草，不强制完成全部设定。
- “分析反馈并调整后文”：`novel-feedback` 归因，再把明确指令交给 `novel-outline` 或 `novel-writing`。

## 路由底线

- 当前意图优先于工作区历史阶段。
- 不在路由层展开正文或深度检查。
- 不把市场指标当作非商业模式的质量门禁。
- 不重复询问项目文件已经能回答的问题。
- 不用规划、检查表或状态文件阻止可逆的探索起草。
