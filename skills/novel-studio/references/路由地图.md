# 路由地图

## 常见用户表达 -> skill

- "帮我开一本新书" -> `novel-project`
- "我有个脑洞，帮我整理一下" -> `novel-project`
- "帮我补人物设定/世界观" -> `novel-worldbuilding`
- "帮我拆成分卷和章节" -> `novel-outline`
- "帮我搭所有卷的卷纲" / "逐卷做分卷规划" -> `novel-outline`
- "帮我细化第X卷的卷纲" / "这一卷写得不够细" -> `novel-outline`
- "帮我检查这章顺不顺、前后有没有打架" -> `novel-quality`
- "继续写下一章" -> `novel-writing`
- "这章太 AI 了，帮我重写" -> `novel-quality`
- "让 AI 帮我写一节对话/战斗/感情戏" -> `novel-ai-writing`
- "让 AI 扮演角色跟我对戏" / "卡文了，让 AI 给我灵感" -> `novel-ai-writing`
- "让 AI 帮我做章节体检/批量生成/风格学习" -> `novel-ai-writing`

## 跨阶段请求

- "从脑洞做到第一卷大纲"
  - 先 `novel-project`
  - 再 `novel-worldbuilding`
  - 先 `novel-outline`（完成总纲和升级阶梯）
  - 再 `novel-outline`（逐卷细化卷纲）

- "把旧稿整理好再继续写"
  - 先 `novel-project`
  - 再根据缺口转 `novel-worldbuilding` 或 `novel-outline`
  - 最后 `novel-writing`

- "我这一章写完了，先帮我查问题再修"
  - 先 `novel-quality`
  - 再转 `novel-quality`

- "让 AI 帮我按节拍卡写一章正文"
  - 先用 `novel-writing` 的写前卡锁定本章任务
  - 再用 `novel-ai-writing` 的"写中-prompt"调用具体技法（对话/战斗/感情/章尾等）
  - 最后用 `novel-ai-writing` 的"写后-prompt"做章节体检

## 更详细的路由

- 平台趋势研究、题材判断 `-> novel-market`
- 拆解爆款/样章/对标作品 `-> novel-market`
- 做书名/简介/钩子/商业化包装 `-> novel-market`
- 去 AI 味、修正模板感 `-> novel-quality`
- 已有全书总纲，逐卷细化卷纲 `-> novel-outline`
- 卷纲已完成，需要拆成章节节拍卡 `-> novel-outline`（卷→章桥接模式）
- 刚填完卷纲骨架，需要逐步细化 `-> novel-outline`（模式一：逐卷推进）
- 想知道这一卷该按什么节奏写（开篇/发展/转折/收束）`-> novel-outline`（读取节奏型模板）
- 想检查多卷的顺序合不合理 `-> novel-outline`（卷序验证）
- 想检查首卷卷纲是否符合开篇节奏 `-> novel-outline`（首卷特殊处理）
- 想检查跨卷钩子的质量 `-> novel-outline`（钩子质量标准）
- 对已有卷纲做局部的调整和修订 `-> novel-outline`
- 让 AI 扮演某角色和我对戏（找灵感/测人物 voice）`-> novel-ai-writing`
- 让 AI 批量生成多章/多节拍卡 `-> novel-ai-writing`
- 让 AI 学习我的文风并模仿 `-> novel-ai-writing`
- 让 AI 做卡文急救 `-> novel-ai-writing`

## 常见跨阶段工作流

### 单章精修链路

> 当用户说"帮我修这一章" / "这章感觉不对" / "读者说这章有问题"

```
定位问题 → 体检诊断 → 制定修复方案 → 执行修订 → 验证修复
```

| 步骤 | 调用 skill | 关键动作 | 产出 |
|------|------------|----------|------|
| 1 定位问题 | novel-studio | 判断用户描述的问题类型（节奏/逻辑/人设/voice/情感） | 问题分类 |
| 2 体检诊断 | novel-quality | 用对应模板（章节体检/逻辑修补/细节修补/回收修补） | 诊断报告 |
| 3 制定方案 | novel-quality | 确定修复范围：局部重写 / 结构调整 / 增加场景 | 修复清单 |
| 4 执行修订 | novel-writing / novel-ai-writing | 按修复清单重写或局部修改 | 修订稿 |
| 5 验证修复 | novel-quality | 重新体检，确认原问题已解决且无新问题 | 验证报告 |

> 如果问题涉及前后多章（伏笔断裂、人设漂移），在第 2 步升级为卷体检。

---

### 读者反馈驱动链路

> 当用户说"读者说这一章XXX" / "评论区有人提XXX" / "追读掉了，帮我看看"

```
收集反馈 → 分类定级 → 定位根因 → 制定改写决策 → 执行改写 → 验证效果
```

| 步骤 | 调用 skill | 关键动作 | 产出 |
|------|------------|----------|------|
| 1 收集反馈 | novel-studio | 引导用户描述读者反馈的具体内容 | 原始反馈记录 |
| 2 分类定级 | novel-studio | 按反馈类型分类（节奏/人设/情感/爽虐/逻辑）并定优先级 | 分类清单 |
| 3 定位根因 | novel-quality | 追溯问题根源：是本章问题还是前文铺垫不足 | 根因分析 |
| 4 制定决策 | novel-quality | 参照 `读者反馈改写决策.md` 确定改写策略 | 决策方案 |
| 5 执行改写 | novel-writing / novel-ai-writing | 按决策执行：局部重写 / 加章 / 调整后续章节 | 改写稿 |
| 6 验证效果 | novel-quality | 检查改写是否解决读者问题，且无新副作用 | 验证报告 |

---

### 设定变更联动链路

> 当用户说"我想加一个新设定" / "这个设定要改" / "战力体系要调整"

```
变更设定 → 评估影响范围 → 更新设定圣经 → 调整大纲 → 重写受影响章节
```

| 步骤 | 调用 skill | 关键动作 | 产出 |
|------|------------|----------|------|
| 1 变更设定 | novel-worldbuilding | 在设定圣经中更新设定，标注变更点和变更原因 | 更新后的设定文件 |
| 2 评估影响 | novel-worldbuilding + novel-outline | 扫描大纲和已写章节，标记所有受影响位置 | 影响范围清单 |
| 3 更新大纲 | novel-outline | 调整受影响卷的卷纲和节拍卡，确保设定一致性 | 更新后的卷纲/节拍卡 |
| 4 重写章节 | novel-writing / novel-ai-writing | 按更新后的大纲重写受影响章节 | 修订稿 |
| 5 一致性验证 | novel-quality | 检查全书设定是否统一，无前后矛盾 | 一致性报告 |

> **重要**：设定变更必须先更新设定圣经和大纲，再重写正文。严禁直接改正文不更新大纲。

### 卡文急救链路

> 当用户说"卡文了" / "不知道接下来怎么写" / "这一章写到一半写不下去"

```
诊断卡文类型 → 选择急救方案 → 快速恢复写作
```

| 卡文类型 | 调用 skill | 急救方案 |
|----------|------------|----------|
| 不知道角色该做什么 | novel-ai-writing（角色扮演） | 让 AI 扮演该角色，问"你现在想做什么？" |
| 不知道冲突怎么推进 | novel-outline | 查阅 `剧情结构骨架库.md`，找对应的推进节点 |
| 情绪调不对 | novel-ai-writing（写中急救） | 用"情绪急救"prompt：先写身体反应，再写对话 |
| 细节不知道怎么写 | novel-writing | 查阅 `50-细节/细节档案.md`，找可复用的细节模板 |
| 节奏乱了 | novel-quality | 用 `写中即时检查卡.md` 快速诊断当前节奏 |
| 完全没灵感 | novel-market | 扫榜找同类型章节的处理方式，激发灵感 |

> 卡文急救以"快速恢复"为第一目标，不求完美，先写出再优化。

### 连载续写链路

> 当用户说"继续写下一章" / "该更新了" / "按大纲往下写"

```
读取当前进度 → 加载写前卡 → 起稿 → 写中检查 → 写后自检 → 更新进度
```

| 步骤 | 调用 skill | 关键动作 | 产出 |
|------|------------|----------|------|
| 1 读取进度 | novel-writing | 读取 `90-运行/当前进度.md` 和本章写前卡 | 写作上下文 |
| 2 确认任务 | novel-writing | 核对本章节拍卡的任务、钩子、情绪目标 | 确认清单 |
| 3 起稿 | novel-writing / novel-ai-writing | 按写前卡执行起稿，优先锁核心戏 | 草稿 |
| 4 写中检查 | novel-writing | 用 `写中即时检查卡.md` 做 mid-draft 质量门禁 | 检查记录 |
| 5 写后自检 | novel-quality | 完成 `章节体检模板.md` 7 项检查 | 体检报告 |
| 6 更新进度 | novel-writing | 回写 `90-运行/当前进度.md`，更新伏笔和因果关系 | 进度文件 |

> 这是日常连载的标准链路。如果体检不通过，走"单章精修链路"而非直接更新进度。

---

### 完结收束链路

> 当用户说"快完结了" / "怎么收尾" / "最后一卷怎么写"

```
评估收束准备度 → 制定收束策略 → 逐卷收束 → 全书体检 → 商业化包装
```

| 步骤 | 调用 skill | 关键动作 | 产出 |
|------|------------|----------|------|
| 1 评估准备 | novel-outline | 检查所有伏笔是否可收、角色弧线是否完整、主线是否闭合 | 收束评估报告 |
| 2 制定策略 | novel-outline | 确定收束节奏：匀速收束 / 加速收束 / 螺旋收束 | 收束策略 |
| 3 逐卷收束 | novel-writing | 按策略写最后几卷，每卷重点检查伏笔回收 | 正文 |
| 4 全书体检 | novel-quality | 做全书级体检：伏笔回收率、人设一致性、结构完整性 | 全书体检报告 |
| 5 商业化包装 | novel-market | 书名/简介/封面文案/平台适配 | 商业化方案 |

> 完结前必须做 `全书一致性验证`，确保没有未回收的伏笔和人设漂移。

---

### 商业化包装链路

> 当用户说"帮我做书名" / "简介怎么写" / "怎么包装更好卖"

```
分析作品卖点 → 对标竞品包装 → 生成书名/简介/钩子 → 平台适配
```

| 步骤 | 调用 skill | 关键动作 | 产出 |
|------|------------|----------|------|
| 1 分析卖点 | novel-market | 提取作品核心卖点（题材+人设+爽点+差异化） | 卖点清单 |
| 2 对标竞品 | novel-market | 扫描同类型爆款的书名/简介/开篇钩子 | 对标分析 |
| 3 生成包装 | novel-market | 按 `商业化写作规范.md` 生成书名/简介/开篇 | 包装方案 |
| 4 平台适配 | novel-market | 根据目标平台调整书名风格和简介长度 | 平台定制版 |

> 商业化包装可在任何阶段进行（立项时/连载中/完结后），越早越好。

## 端到端写作流程（重构模式）

> 当用户说"重构"/"按流程走"/"完整走一遍"/"从头修订"时，启动 7 步强制流程。
> 详细见 `重构流程.md`。

```
Step 1 重审总纲 ──→ Step 2 分卷规划 ──→ Step 3 章节节拍卡 ──→ Step 4 章节写作
                                                               ↓
Step 7 字数定稿 ←── Step 6 去AI味 ←── Step 5 质量检测 ←─────────┘
```

| 步骤 | 调用 skill | 关键产出 | 转入下步条件 |
|------|------------|----------|--------------|
| 1 重审总纲 | novel-outline | 更新后的总纲/升级阶梯/弧线追踪 | 主线能一句话说清 |
| 2 分卷规划 | novel-outline | `分卷/volume-XX.md` | 每卷有功能定位+卷末钩子 |
| 3 章节节拍卡 | novel-outline | `节拍卡/chapter-XXX.md` | 每章有核心戏剧任务+章尾钩子 |
| 4 章节写作 | novel-writing | `30-正文/.../第XXX章-章节名.md` 草稿 | 写后自检通过（含 voice 自检 + 字数在容差内） |
| 5 质量检测 | novel-quality | `40-修订/体检报告/chXXX-checkup.md` | 体检无 P0 + 叙事审计 5 维至少 3 维达标 |
| 6 去AI味 | novel-quality | 修订后的正文 | 去味 5 项自检全过 |
| 7 字数定稿 | novel-writing | 定稿章节 + `90-运行/当前进度.md` 更新 | 字数在容差内 + 状态回写完成 |

> 每步不通过不进入下一步。完整的执行细则见 `重构流程.md`。
